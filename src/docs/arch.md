# Architecture Overview

The Outlook Model Context Protocol (MCP) Server is a modular dual-headed NodeJS application. 

The architecture elegantly separates the **transport layer** (how the AI connects to the app), the **authentication layer** (how the app proves it is the user), and the **service layer** (how the app communicates with external Microsoft servers).

## Dual-Server Design

When initialized, `server.js` inherently binds and spawns two identical but distinct listener platforms.

1. **Express HTTP Server (Port 3000)**
   - Exclusively handles the OAuth 2.0 Webhooks.
   - Operates fully outside of the MCP connection protocol.
   - Provides an endpoint (`/login`) for the user to navigate to on their standard web browser to kickstart MSAL identity resolution.
   
2. **MCP StdIO Transport Platform**
   - Implements the strict `@modelcontextprotocol/sdk` standardized connection.
   - Communicates strictly via Standard Input/Standard Output (STDIO). This is why `console.log` is prohibited (as I/O string streams will corrupt the MCP JSON-RPC protocol messages), and `console.error` is exclusively utilized for terminal debugging.

## Data Flow (How a request reaches Microsoft)

1. **Stdio Event:** AI Agent decides it needs to read an email. It writes a `tools/call` JSON-RPC message into `stdin`.
2. **Schema Validation:** The `@modelcontextprotocol/sdk` inside `server.js` captures the call and delegates `handleToolCall(name, args)` inside `outlookTools.js`.
3. **Graph Invocation:** `outlookTools.js` calls `graphService.fetchEmails()`.
4. **Middleware Intercept:** `graphClient.js` intercepts the outgoing Graph network call, checks the `msalTokenCache.json` for a fresh Access Token using MSAL's `acquireTokenSilent()` and injects the `"Authorization: Bearer XXX"` Header.
5. **Execution:** Result returns up the chain formatted as an array block `[{ type: "text", text: result }]` to `stdout`, terminating the loop seamlessly back to the LLM agent.
